# 决策门禁与验证实验

Use this reference when the user needs a hard startup decision, evidence grading, minimum validation experiment, or stop-loss line.

## 1. Gate Selection

| Gate | Use When | Default Advice |
| --- | --- | --- |
| GO | The core user/value/payment/delivery assumptions have B-level or better evidence, and the next scope is small enough to survive failure. | Start with the smallest controlled version. |
| WAIT | The idea may be valid, but the decisive assumption is still D/C-level or the downside is not understood. | Run a minimum validation experiment before building or signing. |
| KILL | The value is vague, the user is imaginary, the risk is unacceptable, or the founder is using effort to avoid reality. | Stop or archive the idea. |
| PIVOT | The original plan is too heavy, but one smaller part has a believable path. | Change the shape: service before SaaS, manual before automation, niche before broad market. |

Never choose GO just because the user is excited. Never choose KILL just to sound sharp. The gate must follow the evidence.

Evidence grading:

- A: paid order, signed contract, retained usage, verifiable data, formal customer feedback.
- B: user interviews, trial usage, waitlist with clear intent, prototype test, small paid pilot.
- C: public trend, competitor case, social-platform signal, second-hand feedback.
- D: founder belief, friend praise, broad market size, "AI is hot", no direct user evidence.

## 3. Minimum Validation Experiments

Design the experiment around the most dangerous assumption, not the easiest task.

Good experiment:

- Takes 1-7 days when possible.
- Costs little money.
- Produces a clear pass/fail result.
- Uses real users or realistic buyers.
- Tests willingness, behavior, or payment, not just compliments.

Bad experiment:

- Build the full product first.
- Ask friends "do you think this is good?"
- Count likes without a next action.
- Collect vague interest without price, use case, or deadline.
- Make the founder feel productive without reducing uncertainty.

## 4. Stop-Loss Lines

Every serious review should define at least one stop-loss line:

| Risk Type | Stop-Loss Example |
| --- | --- |
| User validation | If fewer than 3 of 10 target users show concrete demand, stop building. |
| Payment | If no one accepts the test price, change offer before development. |
| Time | If the founder spends more than 7 days without external feedback, stop and interview users. |
| Cash | If validation cost exceeds the agreed budget before proof, pause. |
| Compliance | If personal data, contract, payment, or regulated claims cannot be handled safely, do not launch. |
| Delivery | If one-person delivery exceeds the founder's weekly capacity, cut scope or recruit help before selling. |

## 5. Seven-Day Action Plan

When the gate is GO or PIVOT, output:

```text
Day 1: confirm target user and one sentence offer.
Day 2: create prototype, landing note, sample, or manual service flow.
Day 3-4: contact real users, collect objections, ask for payment or commitment.
Day 5: summarize evidence and update scope.
Day 6: decide GO / WAIT / KILL / PIVOT again.
Day 7: either start controlled execution or stop.
```

For WAIT, output only evidence-gathering actions.

For KILL, output archive notes: why killed, what evidence would reopen it, and what not to repeat.
